# parquet-reader

Rust Parquet TUI Viewer。目标是在终端里按页查看 Parquet 文件内容，并保留 Python 原型的核心交互体验。

`parquet_tui.py` 只作为行为参考，不作为 Rust 架构参考。Rust 实现可以重构模块和技术选型，但交互语义必须保持可解释的一致性。

## 常用命令

```bash
cargo fmt
cargo check
cargo test
```

优先用最小可验证命令检查改动。只改文档时不需要强行运行 Rust 编译或测试命令。

## 架构边界

- TUI 层只负责事件处理、布局和渲染；不要直接拼接查询或持有复杂读取逻辑。
- App State 层负责当前视图、分页位置、筛选条件、选中行列、状态栏消息和错误状态。
- Data Access 层负责 Schema、count、分页读取和筛选执行。
- Formatting 层负责 NULL、宽字符、二进制、嵌套结构、超长单元格和终端显示宽度策略。
- 新增功能时优先保持层间边界清晰；不要为快速实现把 TUI、查询和值格式化混在一个函数里。

## 数据读取规则

- 默认按需分页读取 Parquet 数据；除非用户明确要求，不得全量加载文件内容。
- 大文件、宽表、复杂类型和异常 Parquet 元数据都按常见场景处理，而不是边缘场景处理。
- 翻页只移动当前窗口，不改变筛选条件。
- 修改筛选条件后必须重置分页位置。
- 如果总行数统计代价较高，可以设计为懒加载、缓存、可取消或显示未知；不要为了状态栏强制做昂贵全量扫描。
- 用户输入的筛选表达式必须有明确语义边界；如果透传给查询引擎，UI 和文档都必须说明它不是安全沙箱。
- 筛选失败、读取失败和空结果都不得导致程序崩溃；保留上下文并展示可行动错误。

## TUI 交互规则

- 快捷键保持 vim/k9s 风格。
- 已有导航习惯不得被新功能破坏：`h/j/k/l`、方向键、翻页、Schema 切换、筛选、重置和退出语义要稳定。
- 常用操作必须能只靠键盘完成。
- 筛选输入、错误提示和临时状态不要永久遮挡用户正在查看的数据。
- 保持跨平台终端兼容；不要依赖特定 shell、字体、终端模拟器或操作系统私有行为。

## 值显示与错误处理

- 单元格显示必须稳定处理 NULL、超长文本、宽字符、二进制、列表、Map、Struct 等值。
- 超长单元格要截断并显示省略标记；截断不得破坏 Unicode 边界。
- 单个异常值不得破坏整张表格布局。
- 面向终端用户的错误信息要短、明确、可行动。
- 内部错误链可以保留给日志或调试输出，但不要默认把完整数据行写入日志。
- 失败路径和空数据路径是一等测试场景，不要只验证 happy path。

## 依赖与实现约束

- 新增依赖前先说明理由。
- 避免引入与 TUI、Parquet 读取、CLI、错误处理或测试无关的重量级依赖。
- 数据访问实现可以选择 Arrow、DataFusion、DuckDB 绑定或其他方案，但必须被封装在 Data Access 边界内。
- 不要让具体查询引擎的表达式、异步模型或错误类型泄漏到 TUI 层。
- 小步提交、小步验证；每次代码改动至少能通过格式化、编译或测试中的一种方式验证。

## 文档维护规则

- 设计文档、需求文档和代码实现要同步更新。
- 文档描述尚未落地的行为时，必须明确标注为规划或待决策。
- `AGENTS.md` 只写 agent 长期需要遵守的规则；不要写目录树、依赖版本、文件清单等可通过简单查看或搜索得到的信息。
- 面向人类的新手说明、安装教程、完整功能介绍应放在 README 或 docs 中，不要堆进 `AGENTS.md`。

## 禁止行为

- 禁止默认全量读取 Parquet 文件来简化分页、筛选或统计。
- 禁止把 TUI 渲染、应用状态、数据查询和值格式化混成无边界实现。
- 禁止把筛选表达式描述成安全沙箱。
- 禁止提交真实业务数据、隐私数据、大型样例 Parquet 文件或本机绝对路径。
- 禁止提交 `target/`、临时文件、编辑器缓存、性能采样产物或生成日志。
- 禁止把 Python 原型的代码结构直接照搬为 Rust 架构。
