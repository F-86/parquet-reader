# docs/design.md 重构计划

## Brief

- 文档类型：设计 / 架构文档
- 场景：方案设计 / RFC
- 目标受众：未来贡献者、维护者、AI agent
- 核心目标：解释 Rust Parquet TUI Viewer 为什么按分层架构实现、第一阶段做什么、不做什么、关键技术取舍在哪里。
- 必须保留：
  - Python 原型的核心交互参考。
  - 分页读取和禁止默认全量加载。
  - 数据视图、Schema 视图、筛选、状态栏、快捷键。
  - TUI / App State / Data Access / Formatting 分层。
  - Arrow / DataFusion / DuckDB 三条数据访问路线对比。
  - 错误处理、测试计划、里程碑、待决策项。
- 可删减：
  - 重复需求描述。
  - 过细但未决的实现细节。
  - 读者可从源码直接获得的目录事实。
- 语气风格：设计评审文档，短段落，强调决策和边界。
- 目标语言：中文。

## Outline

1. `# Parquet TUI Viewer 设计方案`
   - 标明这是拟议方案，不是已完成架构现状。
2. `## 1. 背景`
   - 说明项目目标和 Python 原型关系。
3. `## 2. 目标与非目标`
   - 明确第一阶段能力边界。
4. `## 3. 用户体验需求`
   - 压缩整理 CLI、数据视图、Schema、筛选、导航、状态栏。
5. `## 4. 核心约束`
   - 大文件、分页、只读、安全边界、终端兼容。
6. `## 5. 整体架构`
   - 使用 Mermaid flowchart 表达 CLI、App State、TUI、Data Access、Formatting 的关系。
7. `## 6. 模块职责`
   - 用表格说明模块职责、输入输出、禁止事项。
8. `## 7. 应用状态与动作模型`
   - 保留并强化 `Action -> State transition -> DataCommand -> Render`。
9. `## 8. 关键流程`
   - 增加启动、翻页、筛选、Schema 切换流程。
10. `## 9. 数据访问设计`
    - 定义 trait 边界、分页、count、筛选表达式语义。
11. `## 10. 技术选型`
    - TUI 选型明确；数据访问路线对比并保留待决策。
12. `## 11. 显示、错误与边界场景`
    - 统一值格式化、错误分类、空数据和异常路径。
13. `## 12. 测试策略`
    - 单元、集成、手动验收按风险组织。
14. `## 13. 里程碑`
    - 每个里程碑写交付能力和验收标准。
15. `## 14. 风险与待决策项`
    - 合并已知风险、缓解策略、开放问题。

## 特殊注意事项

- 设计文档不要写成 README，不写安装教程。
- 不虚构当前代码已实现的功能；未实现能力用「第一阶段建议」或「待决策」表述。
- Mermaid 图用于架构和数据流；目录树仍可用 text 代码块。
- 技术选型如无法最终决定，应明确“推荐下一步评估”而不是伪装成已确定。

## Plan 自查

1. 目标受众明确：贡献者、维护者、AI agent。
2. 信息范围明确：重构设计方案，不改代码，不写 README。
3. 覆盖用户重点：Rust、Parquet TUI、Python 原型、架构和需求。
4. 章节补齐：目标/非目标、架构图、关键流程、风险。
5. 计划可读：每节说明了目的和迁移方向。
