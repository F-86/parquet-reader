# docs/design.md 诊断结果

## 文档类型

- 类型：设计 / 架构文档
- 场景：方案设计 / RFC，描述拟议中的 Rust Parquet TUI Viewer，而不是已建成系统现状
- 目标受众：未来贡献者、维护者、AI agent
- 修改模式：重构 + 规范化

## 🔴 严重（与设计文档规范冲突）

- [ ] 缺少明确的「目标与非目标」章节。
  - 建议：将当前需求、第一阶段范围、禁止项重排为「目标」「非目标」，先划边界再讲架构。

- [ ] 缺少 Mermaid 架构图。
  - 建议：把现有 ASCII 分层图改为 Mermaid flowchart；保留目录树可以继续使用 text 代码块。

- [ ] 「需求」和「架构」混排，读者需要先读很长需求才能看到系统为什么这样设计。
  - 建议：重排为 RFC 风格：背景 → 目标/非目标 → 用户体验 → 架构 → 数据流 → 关键决策 → 技术选型 → 测试与里程碑。

- [ ] 技术选型建议给了路线，但没有把“推荐方案”和“待决策项”清晰分离。
  - 建议：明确当前推荐：TUI 使用 `ratatui` + `crossterm`；数据访问路线仍待决策，或给出阶段性推荐。

## 🟡 改进（偏离最佳实践）

- [ ] 部分章节重复表达分页、大文件、筛选错误等约束。
  - 建议：把重复内容归并到「核心约束」和「数据访问设计」。

- [ ] 当前架构章节偏“模块清单”，缺少关键流程。
  - 建议：增加启动流程、翻页流程、筛选流程、Schema 切换流程。

- [ ] 「状态转移」内容有价值，但位置偏深。
  - 建议：提升为「应用状态与动作模型」，让 TUI 实现时更容易落地。

- [ ] 错误处理、测试计划和里程碑都存在，但与需求追踪关系不清。
  - 建议：按功能域整理，让测试项对应到架构边界和关键风险。

- [ ] 文档标题是「设计文档」，但内容包含大量需求。
  - 建议：保留需求，但压缩为「产品需求摘要」；详细需求后续可拆到独立 PRD 或 requirements 文档。

## 🟢 建议（锦上添花）

- [ ] 增加「术语表」。
  - 建议：定义 Page、Offset、Filter、Schema View、Data Access 等术语，减少后续实现歧义。

- [ ] 增加「演进路线」而不是只列 M1-M4。
  - 建议：把每个里程碑绑定交付能力和验收标准。

- [ ] 增加「风险与缓解」。
  - 建议：覆盖大文件 count、终端宽字符、查询引擎依赖、异步阻塞 UI 等风险。

## 修复日志

- 将文档重构为方案设计 / RFC 风格。
- 增加「目标与非目标」。
- 将 ASCII 架构图改为 Mermaid 架构图。
- 增加启动、翻页、筛选、Schema 切换关键流程。
- 将模块职责整理为表格，并明确每层禁止事项。
- 将状态模型和动作模型提升为独立章节。
- 将技术选型拆分为 TUI 推荐和 Parquet 读取路线对比。
- 增加显示策略、错误分类、边界场景、风险与缓解。
- 将里程碑改为交付能力 + 验收标准。
