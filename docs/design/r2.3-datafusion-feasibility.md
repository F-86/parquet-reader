# R2.3 DataFusion 可行性调研

> 状态：设计结论（不落代码）
> 日期：2026-07-15

## 一、是否值得引入

**当前结论：暂不引入。** 在当前功能需求和文件规模下，引入 DataFusion 的收益不足以抵消其成本。

### 判断依据

当前实现已经具备的能力：

- 分页读取（row-group aware，跳过不需要的 row group）
- 受限 DSL 筛选（`and`/`or`，7 种操作符）
- 类型化比较（数字/布尔/字符串/Null）
- 列级最小化匹配（只提取筛选引用列的 typed value）
- 手动 count（按需扫描，不阻塞 UI）

DataFusion 能带来但当前**尚未需要**的能力：

- SQL 表达式（括号优先级、函数、子查询）
- Parquet predicate pushdown（row group 级别统计信息过滤）
- 列裁剪下推到 Parquet reader
- 优化执行计划（filter + limit + projection 融合）

### 不引入的理由

1. **当前 DSL 已满足核心需求**：用户主要做 `column op value` 的简单筛选，`and`/`or` 组合已覆盖大多数场景。
2. **predicate pushdown 的实际收益有限**：当前 row-group-aware pagination 已经跳过不相关的 row group；pushdown 在 row group 内部按统计信息跳过，收益取决于数据排序和 row group 大小，对随机数据收益接近零。
3. **依赖成本显著**：`datafusion` 会引入完整的 SQL parser、逻辑/物理优化器、异步执行 runtime，编译时间增加数分钟级，二进制体积增加数十 MB。
4. **异步模型冲突**：DataFusion 基于 `tokio` 异步执行，当前 TUI 是同步事件循环；引入异步需要在 Data Access 边界做 sync→async 桥接，增加复杂度。

### 何时重新评估

以下任一条件成立时值得重新评估：

- 需要支持括号优先级或复杂嵌套表达式
- 需要在筛选中使用聚合函数（如 `sum(score) > 1000`）
- 文件规模达到 GB 级且数据按筛选列有序存储（pushdown 收益显著）
- 需要 SQL 兼容性以支持外部工具集成

## 二、会带来哪些收益

| 收益 | 描述 | 当前是否需要 |
|------|------|-------------|
| Predicate pushdown | 利用 row group min/max/bloom filter 跳过不匹配的 row group | 低——当前 row-group-aware pagination 已跳过整组 |
| 列裁剪下推 | Parquet reader 层面只读取筛选涉及列 | 中——当前已在内存层面做列级最小化匹配，但 reader 层面仍读取全部列 |
| 复杂表达式 | 支持括号、嵌套 `and`/`or`、函数调用 | 低——当前 DSL 已支持 `and`/`or` 优先级 |
| 优化执行计划 | filter→limit→projection 融合，减少中间数据 | 低——当前实现已经是 streaming + early-exit |
| Count 优化 | 利用 metadata 统计信息快速估算 count | 中——当前 count 需要全量扫描 |
| 未来扩展性 | 可叠加 JOIN、UDF、窗口函数等 | 低——TUI viewer 不需要这些 |

**结论**：列裁剪下推和 count 优化是唯一有实际价值的收益，但可以通过更轻量的方式实现（如直接使用 `ParquetRecordBatchReaderBuilder::with_row_groups` + 列投影），不需要引入完整 DataFusion。

## 三、会破坏哪些当前架构边界

### 3.1 异步模型泄漏

DataFusion 基于 `async` 执行。当前架构：

```
TUI (sync event loop) → AppState::handle_action (sync) → DataCommand → DataSource (sync)
```

引入 DataFusion 后需要在 Data Access 层引入 async runtime。根据 AGENTS.md 约束："不要让具体查询引擎的异步模型泄漏到 TUI 层"。这意味着需要在 Data Access 边界做 `block_on` 或 channel 桥接，增加复杂度和潜在死锁风险。

### 3.2 错误类型泄漏

DataFusion 的 `DataFusionError` 枚举包含 SQL parse error、plan error、execution error 等大量变体。根据 AGENTS.md 约束："不要让具体查询引擎的错误类型泄漏到 TUI 层"。需要在 Data Access 边界做完整错误映射，维护成本高。

### 3.3 表达式语法泄漏

如果允许用户输入 SQL 表达式，需要根据 AGENTS.md 约束在 UI 和文档中明确标注"表达式语法来自底层引擎，不是安全沙箱"。这与当前受限 DSL 的清晰语义边界形成冲突——用户需要学习 SQL 语法而非简洁的 `column op value` DSL。

### 3.4 依赖膨胀

`datafusion` 拉入 `tokio`、`sqlparser`、`parking_lot`、`async-trait` 等数十个间接依赖。根据 AGENTS.md 约束："避免引入与 TUI、Parquet 读取、CLI、错误处理或测试无关的重量级依赖"。

## 四、如果引入，边界应如何封装

假设未来决定引入，以下是推荐的封装策略：

### 4.1 Data Access trait 保持不变

```rust
// 当前 trait 签名不需要改变
pub fn read_page_with_filter(&self, offset: usize, limit: usize, filter: Option<&str>) -> Result<DataPage>
pub fn count_with_filter(&self, filter: Option<&str>) -> Result<usize>
```

DataFusion 实现作为 `ParquetFileDataSource` 的一个内部 backend，对外接口完全不变。

### 4.2 DSL → DataFusion Expr 转换层

在 Data Access 层内部新增一个 `dsl_to_datafusion` 模块，将当前 `FilterAst` 转换为 DataFusion `Expr`。用户输入仍然是受限 DSL，不暴露 SQL。

```
User DSL → FilterAst → DataFusion Expr → ExecutionContext::filter
```

### 4.3 异步隔离

在 Data Access 层内部创建一个专用 `tokio::runtime::Runtime`，通过 `runtime.block_on()` 调用 DataFusion。TUI 层不感知 async。

### 4.4 错误映射

`DataFusionError` 映射为 `AppError::InvalidFilter` 或 `AppError::DataAccess`，丢弃内部细节，只保留面向用户的短消息。

### 4.5 Feature gate

通过 Cargo feature flag 控制：

```toml
[features]
datafusion-backend = ["dep:datafusion"]
```

默认编译使用原生 Arrow 读取，仅在需要时启用 DataFusion 后端。

### 4.6 列裁剪先行

即使不引入 DataFusion，也可以在当前实现中做轻量列裁剪：使用 `ParquetRecordBatchReaderBuilder` 的列投影能力，只读取筛选引用的列 + 需要显示的列。这是当前架构内可实现的优化，不需要引入任何新依赖。

## 五、总结

| 维度 | 结论 |
|------|------|
| 是否引入 | 暂不引入 |
| 主要收益 | 列裁剪下推、count 优化 |
| 主要成本 | 依赖膨胀、异步桥接、错误映射 |
| 替代方案 | 轻量列裁剪（原生 Arrow），不引入新依赖 |
| 重新评估触发条件 | 需要复杂表达式、GB 级有序文件、SQL 兼容性 |

建议下一步优先实现 **轻量列裁剪**（在不引入 DataFusion 的前提下，在 reader 层面只读取需要的列），这是当前架构内的增量优化，收益与 DataFusion 列裁剪下推基本等价。
